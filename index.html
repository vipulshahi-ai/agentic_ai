What is Agentic AI?

Before we build agents, let's understand what they actually are.

🤖

From Chatbots to Agents

You've used ChatGPT. You type, it replies. But what if AI could take actions on its own — browse the web, write code, book flights, send emails — without you babysitting every step?
That's the leap from AI to Agentic AI.

What is an AI Agent?

An AI Agent is a software program that can perceive its environment, reason about what to do, and take actions to achieve a goal — all on its own.

Think of it like the difference between a calculator and a personal assistant:

🔢
Traditional AI (Calculator)

You ask a question, it gives one answer. Done. No follow-up, no actions.

💼
AI Agent (Personal Assistant)

You give it a goal, it plans steps, uses tools, checks results, and loops until the job is done.

What is Agentic AI?

Agentic AI is the design pattern of building AI systems that behave like agents — systems that can autonomously decide, act, and iterate to accomplish complex tasks.

Why did the LLM get it wrong?

The LLM processes text as tokens (word-pieces), not as numbers. When it sees "34567 × 89123", it doesn't multiply — it predicts what digits would likely follow based on patterns in its training data. It's essentially guessing, not calculating!

🧩
The Core Limitation

LLMs are word-predictors, not calculators. They excel at language tasks but fail at precise computation, live data access, and real-world actions. This is exactly why AI Agents need Tools!

🔍 The "Stochastic Parrot" Problem: The LLM generates statistically plausible text — not mathematically correct text. For every digit it outputs, it's asking: "What digit would most likely come next?" — not "What is the actual product?"
💡 Key Insight: This is exactly why agents need Tools. The Brain is smart at language, but dumb at math. Solution? Give it a calculator!

🏷️ Understanding @tool — What Is It & Why Do We Need It?

Before we write the code, let's understand the most important line in the entire tool system:

@tool

This single line transforms a normal Python function into an agent-usable tool.

But what IS a decorator?

In Python, a decorator is a special annotation you place above a function. It wraps your function with extra superpowers — without changing your original code.

📦
Without @tool

Your function is a plain Python function. Only you can call it. The LLM doesn't know it exists.

🏷️📦
With @tool

Your function gets a label (name + description). Now the LLM can discover it, understand when to use it, and request it!

Real-world analogy: The Labeled Toolbox

Imagine you have a toolbox full of tools, but none of them have labels. You hand the toolbox to a new intern.

🧰
Intern (LLM) opens the toolbox...

Sees a bunch of tools but has no idea what they do. Useless!

⬇️ Now you add labels to each tool...
🏷️🔧
Labeled tools!

"Hammer — Use to drive nails into wood."
"Screwdriver — Use to tighten or loosen screws."
Now the intern knows exactly which tool to pick for each job!

🔍 So what does @tool actually do? Three things:
1 Registers the name

The function name (multiply) becomes the tool's name that the LLM sees.

2 Reads the docstring

The """Multiply two numbers""" becomes the tool's description — tells the LLM when to use it.

3 Extracts the parameters

The type hints (a: float, b: float) tell the LLM what inputs the tool needs.

The Complete Code — From Tool to Answer

Here's the entire code that defines a tool, binds it to the LLM, asks a question, and executes the tool call manually:

Python — The Complete Tool Pipeline
from langchain_core.tools import tool @tool def multiply(a: float, b: float) -> float: """Multiply two numbers together.""" return a * b # Hand the calculator to the student llm_with_tools = llm.bind_tools([multiply]) response = llm_with_tools.invoke(question) # The LLM doesn't answer directly — it asks to use the tool! print(f"Content: {response.content}") print(f"Tool Calls: {response.tool_calls}") # YOU are the Runtime here! tool_calls = response.tool_calls if tool_calls: action = tool_calls[0] # Get the request func_name = action['name'] # "multiply" args = action['args'] # {'a': 34567, 'b': 89123} if func_name == "multiply": result = multiply.func(args['a'], args['b']) print(f"Calculated Answer: {result}") else: print("AI answered directly:", response.content)
📌 This code has 3 phases:
Phase 1 (Lines 1-6): Define the tool using @tool
Phase 2 (Lines 8-14): Bind the tool to the LLM and ask the question
Phase 3 (Lines 16-26): Execute the tool call manually (YOU are the runtime!)

LangGraph — The Autopilot 🔄

Automate the agent loop with State, Nodes, and Edges.

But First — What is LangGraph?

In Level 1, we learned that LangChain gives us building blocks — LLM connectors, tools, messages. But who orchestrates them? Who decides "call the LLM → check if it needs a tool → run the tool → go back to the LLM → repeat"?

That's exactly what LangGraph does.

🍳
LangChain = Ingredients

Flour, eggs, butter, sugar — all the individual pieces you need. You still have to mix them yourself.

📜
LangGraph = The Recipe

Step 1: Mix dry. Step 2: Add wet. Step 3: Check consistency. If lumpy → go back to Step 2. Done → bake!

📚 In short: LangGraph is a Python library (by the LangChain team) that lets you build agent workflows as graphs — with nodes (workers), edges (connections), and state (shared data). It automates the "think → act → observe → repeat" loop.

LangGraph's 3 Core Concepts

LangGraph models everything as a graph — like a flowchart. It has just 3 building blocks:

📋
State

A shared whiteboard that all nodes can read and write. It carries data (messages, results) between steps.

⚙️
Nodes

Workers on the assembly line. Each node does ONE job — "call the LLM" or "run the tool". They're just Python functions.

➡️
Edges

The conveyor belts connecting workers. They define the flow: "after LLM, go to Router. If tool needed, go to Tools. If done, go to END."

Now let's implement each concept in code — one at a time!

📋 Code Section 1: Defining the State (The Shared Clipboard)

Before our graph can do anything, we need a shared clipboard that all workers (nodes) can read from and write to. In LangGraph, this clipboard is called the State.

But what IS State, really?

Think of it this way: In Level 2, when the LLM sent a message and the tool sent back a result — where did those messages live? They were just floating around in variables. There was no central place to store the conversation.

State fixes that. It's a Python dictionary that holds all the messages — the user's question, the LLM's replies, the tool results — in one place. Every node reads from it and writes back to it.

📋
State = Clipboard

A shared clipboard passed between workers on a factory line. Worker 1 writes, Worker 2 reads, Worker 2 writes, etc.

📝
add_messages = Append Mode

"Don't erase what's on the clipboard — just add a new sticky note below!" New messages get appended, never overwritten.

Python — Define the State
from typing import Annotated from typing_extensions import TypedDict from langgraph.graph.message import add_messages class State(TypedDict): messages: Annotated[list, add_messages] # Translation: "State has a 'messages' list. # When new info comes, ADD it, don't overwrite."
📌 Line-by-line:
  • TypedDict → A Python way to create a dictionary with defined keys. Like saying "this dict MUST have a key called 'messages'."
  • Annotated[list, add_messages] → This is the magic part! It says: "messages is a list, and when you update it, use add_messages (append, don't replace)."
  • add_messages → A LangGraph helper that intelligently appends messages. If you return {"messages": [new_msg]}, it gets added to the existing list, not overwritten.

🤔 Why Annotated? Why not just list?

Without Annotated, every time a node returns {"messages": [something]}, it would replace the entire messages list — wiping out the conversation history! With add_messages, it appends instead, preserving the full conversation.

❌
Without add_messages (Replace mode)

Clipboard: ["Hi"] → Worker returns ["The answer is 42"] → Clipboard: ["The answer is 42"] — original question LOST!

vs
✅
With add_messages (Append mode)

Clipboard: ["Hi"] → Worker returns ["The answer is 42"] → Clipboard: ["Hi", "The answer is 42"] — full history preserved!

⚙️ Code Section 2: Creating the Nodes (The Workers)

Now that we have a clipboard (State), we need workers to actually do things. In LangGraph, workers are called Nodes. Each node is just a Python function that reads the State, does something, and writes back.

Our agent needs exactly 2 workers:

🧠
Worker 1: The Assistant

Reads messages from the clipboard → sends them to the LLM → writes the LLM's response back. This is the "thinking" worker.

🛠️
Worker 2: The Tool Runner

Reads the LLM's tool request from the clipboard → actually runs the tool (e.g., multiply) → writes the result back. This is the "doing" worker.

Python — Create the Nodes (Workers)
from langgraph.prebuilt import ToolNode # Worker 1: The Assistant (asks LLM to think) def assistant_node(state: State): response = llm_with_tools.invoke(state["messages"]) return {"messages": [response]} # Worker 2: The Tool Runner (executes tool calls) tool_node = ToolNode([multiply])
📌 Line-by-line:
  • assistant_node(state: State) → A regular Python function that takes the State as input.
  • state["messages"] → Reads all messages from the clipboard (conversation history).
  • llm_with_tools.invoke(state["messages"]) → Sends the entire conversation to the LLM (which has tools bound from Level 2).
  • return {"messages": [response]} → Writes the LLM's reply back to the clipboard. Thanks to add_messages, this appends, not replaces!
  • ToolNode([multiply]) → A pre-built LangGraph node that automatically knows how to execute tool calls. No manual if/else needed — it handles everything from Level 3 automatically!
💡 Notice: ToolNode replaces all the manual if/else code from Level 3! Remember those 10+ lines of extracting func_name, args, and calling multiply.func()? ToolNode does it all in one line. That's the power of LangGraph automation!

➡️ Code Section 3: The Router Edge (The Traffic Signal)

We now have a clipboard (State) and two workers (Nodes). But how does the system know which worker to go to next? That's the job of Edges — specifically, conditional edges that act like traffic signals.

🚦
The Router is a traffic signal at a crossroad:

After the LLM (Assistant) responds, the router checks: "Did the LLM ask for a tool?"

🟢 Yes → "tools" → Send the message to the Tool Runner
🔴 No → END → The LLM has the final answer, stop the loop

Python — The Router (should_continue)
from langgraph.graph import END def should_continue(state: State): last_msg = state["messages"][-1] if last_msg.tool_calls: # LLM wants a tool? return "tools" # → Go to tool node return END # → Done, stop the loop
📌 Line-by-line:
  • state["messages"][-1] → Gets the last message on the clipboard (the LLM's most recent reply).
  • last_msg.tool_calls → Checks: does this message contain tool call requests? (Remember from Level 2 — when the LLM wants the calculator, tool_calls is non-empty!)
  • return "tools" → Tells LangGraph: "Send the state to the node named 'tools' next."
  • return END → Tells LangGraph: "We're done! The LLM's answer IS the final answer. Stop the loop."

🔄 Why is this a loop?

Because after the Tool Runner executes the tool and writes the result back, the graph will send it back to the Assistant. The Assistant sees the tool result, generates a human-readable final answer, and the router checks again — this time, tool_calls is empty, so it goes to END.

The Code Behind the Graph

Python — Build & Compile Graph
from langgraph.graph import StateGraph, START, END builder = StateGraph(State) builder.add_node("assistant", assistant_node) builder.add_node("tools", tool_node) builder.add_edge(START, "assistant") # Start → Assistant builder.add_conditional_edges( # Router logic "assistant", should_continue ) builder.add_edge("tools", "assistant") # Tools → back graph = builder.compile()
📌 Line-by-line:
  • StateGraph(State) → We initialize the graph by giving it our State class. This tells the graph the format of its shared clipboard.
  • builder.add_node(...) → We register our workers as nodes in the graph, giving them string names like "assistant" and "tools".
  • builder.add_edge(START, "assistant") → We draw a hardcoded connection from the start line. The assistant always goes first.
  • builder.add_conditional_edges(...) → We install our traffic light (should_continue router). After the assistant thinks, this dynamically checks whether to route to the tools or end the loop.
  • builder.add_edge("tools", "assistant") → We wire the tools to always dump their results back to the assistant.
  • builder.compile() → We hit "save". This freezes the nodes and edges, compiling everything into a single, highly optimized, executable app.

🚀 Code Section 4: Running the Graph (The Magic Button)

We've defined our State, built our Nodes, wired our Edges, and compiled the graph. Now it's sitting idle, waiting for work. Let's flip the switch and watch the autopilot run a multi-step math problem!

Python — Execute the Agent
query = "Calculate 50 times 173. Then divide that result by 5." result = graph.invoke({"messages": [("user", query)]}) print(result["messages"][-1].content)
📌 Line-by-line:
  • graph.invoke(...) → This single command kicks off the entire automated loop!
  • {"messages": [("user", query)]} → We are placing the initial state onto the clipboard. We give it one message from the user.
  • result → This contains the final state of the clipboard once the graph hits END. It has the full conversation history (User → LLM asks for tool → Tool answers → LLM answers → etc).
  • result["messages"][-1].content → We fetch the very last message on the clipboard (which is the LLM's final human-readable answer) and print its text.
🤯 Why is this mind-blowing? With one invoke() call, LangGraph handled: 1. Sending the prompt. 2. Parsing the tool request for multiplication. 3. Running the math tool. 4. Sending the result back to the LLM. 5. Reading the division instruction. 6. Running the division tool. 7. Generating the final answer! Everything we had to manually code in Level 3 is now running on Autopilot.

🏗️ Step 4: Automating the Flow with LangGraph

To handle multiple tool calls sequentially (fetch price, then multiply, then answer), we need to build our LangGraph Autopilot. We define the State, the Workers, the Router, and finally compile it all!

Python — The State (Clipboard)
from typing import Annotated from typing_extensions import TypedDict from langgraph.graph.message import add_messages class State(TypedDict): messages: Annotated[list, add_messages]
Python — Workers (Nodes) & Router (Edges)
from langgraph.prebuilt import ToolNode from langgraph.graph import END tools = [get_crypto_price, multiply] # Worker 1: The Assistant (asks LLM to think) def assistant_node(state: State): response = llm_with_tools.invoke(state["messages"]) return {"messages": [response]} # Worker 2: The Tool Runner (executes tool calls) tool_node = ToolNode(tools) # The Router: Traffic light logic def should_continue(state: State): last_msg = state["messages"][-1] if last_msg.tool_calls: # LLM wants a tool? return "tools" # → Go to tool node return END # → Done, stop the loop
Python — Assemble and Compile
from langgraph.graph import StateGraph, START, END builder = StateGraph(State) builder.add_node("assistant", assistant_node) builder.add_node("tools", tool_node) builder.add_edge(START, "assistant") # Start → Assistant builder.add_conditional_edges( # Router logic "assistant", should_continue ) builder.add_edge("tools", "assistant") # Tools → back graph = builder.compile()
📌 Line-by-line:
  • We recreate our trusty State to hold the conversation history.
  • We define assistant_node to call the LLM, and ToolNode to automatically map any tool requests to our tools list.
  • Our should_continue router acts as a traffic cop. If the LLM produces a tool call, we divert to the tools. Otherwise, we stop at END.
  • Finally, we wire our nodes and edges together on the StateGraph and compile() it to lock it in!

📌 Connecting the Concepts:
  • Assistant Node 🤖: The "Brain" of the operation. It assesses the conversation history and decides if it can answer right away or if it needs to trigger a tool.
  • Conditional Router 🚦: Automatically inspects the Assistant's output. If it sees a tool request, it routes execution DOWN to the Tools Node. If it sees a final answer, it routes RIGHT to end the sequence.
  • Tools Node 🛠️: Executes any requested Python tools (like fetching Crypto Prices or Multiplying). When it's done, it ALWAYS loops forcefully back up to the Assistant to feed it the new data.

🌐 Step 1: Initialize the Brain & State

First, we need a powerful LLM to act as the core brain for all our agents, and a shared whiteboard (State) to pass information between them.

Python — LLM & State
from langchain_groq import ChatGroq from typing import TypedDict llm = ChatGroq( model="llama-3.3-70b-versatile", temperature=0, # 0 = factual, 1 = creative api_key="YOUR_API_KEY" ) class SoftwareState(TypedDict): user_request: str # What the user wants to build requirements: str # PM fills this with the spec code: str # Developer fills this with code qa_feedback: str # QA fills this with review notes iteration_count: int # Tracks how many Dev-QA cycles status: str # "APPROVED"
📌 Why does this matter? Notice how our SoftwareState has specific fields for each agent's job! The PM touches requirements, the Developer touches code, and QA touches qa_feedback. They all share this dictionary as their sole means of communication.

👨‍💼 Step 2: Define the Worker Agents (Nodes)

Instead of tools, the nodes in this graph are distinct sub-agents. Each agent runs the LLM with a specific prompt tailored to its "role" (PM, Dev, QA).

Python — PM, Dev, and QA Nodes
def pm_node(state): print("[PM] Analyzing request...") prompt = f'Create a technical spec for: "{state["user_request"]}"' response = llm.invoke(prompt) print(f"[PM] Spec ready! ({len(response.content)} chars)") return {"requirements": response.content} def dev_node(state): iteration = state["iteration_count"] + 1 if state.get("qa_feedback"): print(f"[DEV] Rewriting code (attempt #{iteration})...") context = f"PREVIOUS CODE REJECTED. FEEDBACK: {state['qa_feedback']}. Fix it." else: print(f"[DEV] Writing initial code (attempt #{iteration})...") context = "" prompt = f"Write Python code for: {state['requirements']}. {context}" response = llm.invoke(prompt) print(f"[DEV] Code written! ({len(response.content)} chars)") return {"code": response.content, "iteration_count": iteration} def qa_node(state): print("[QA] Reviewing code...") prompt = f"Review this code: {state['code']}. Say APPROVED or REJECTED with reasons." response = llm.invoke(prompt) status = "APPROVED" if "APPROVED" in response.content else "REJECTED" print(f"[QA] Verdict = {status}") return {"qa_feedback": response.content, "status": status}

🚦 Step 3: The Manager (Router)

We need a routing conditional edge to automate the back-and-forth approval loop between the Developer and QA agent.

Python — The QA Loop Router
def router(state): if state["status"] == "APPROVED": return END # QA approved! Stop the loop. if state["iteration_count"] > 3: return END # Safety: max 3 attempts return "developer" # Rejected! Send back to Dev.
📌 Notice: We added an iteration_count > 3 safety catch. In multi-agent systems, agents can get stuck endlessly arguing (e.g., Code fails → Dev rewrites → Code fails → Dev rewrites). Always add circuit breakers!

🏗️ Step 4: Assemble the Multi-Agent Flow

Wire all the workers together along with our QA loop conditionally determining if the workflow ends or continues.

Python — Build the Graph
from langgraph.graph import StateGraph, START, END workflow = StateGraph(SoftwareState) # Register the 3 agents as nodes (workers) workflow.add_node("pm", pm_node) workflow.add_node("developer", dev_node) workflow.add_node("qa", qa_node) # Draw the arrows (edges) between them workflow.add_edge(START, "pm") # Entry point: PM goes first workflow.add_edge("pm", "developer") # PM's output goes to Dev workflow.add_edge("developer", "qa") # Dev's output goes to QA # QA's output goes to the Router (conditional) workflow.add_conditional_edges( "qa", router, {"developer": "developer", END: END} ) app = workflow.compile() print("Software House ready!")

📌 Connecting the Concepts:
  • State 📋: The invisible clipboard flowing along the arrows! It carries the user_request, code, and qa_feedback between workers.
  • Nodes 🔵: The actual worker stations (pm_node, dev_node, qa_node). This is where the LLM does the thinking and ultimately updates the State.
  • Normal Edges ⬇️: Fixed conveyor belts. The code will always blindly flow from PM to Developer, and from Developer to QA.
  • Conditional Edges 🚦: The traffic cop! Exiting the Router, the flow isn't fixed. It evaluates the State dynamically (e.g., status == "APPROVED") to decide which path to take!
